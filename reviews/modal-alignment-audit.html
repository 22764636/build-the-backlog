Modal Alignment Audit
Every multi-field row in the Add/Edit modal (#mov, including the embedded Collection Details section) and the Move-to-Collection modal (#btcov), measured with getBoundingClientRect() at 1280×1000 @2x DPR. For each row: do all labels end (bottom edge) at the same Y, and do all fields start (top edge) at the same Y? Field height is allowed to differ when the controls are genuinely different types (e.g. a date-overlay widget vs. a plain input) — see the Release Date row below for a passing example of that exception.
Report only — no fixes applied yet. Measurements taken via Playwright (Chromium, 1280×1000 viewport, 2x device scale factor), seeding a wishlist game and opening each modal directly.

The unpaired <sup> grows one label to 15.36px against the other's plain 11px, producing the same +4.36px offset on both label-bottom and field-top.
Fix, not yet applied: both failing rows share one mechanism — a lone <sup> growing one label's line-box while its row partner stays at plain-text height. The existing fix (.fg3-mid-stretch .fr label{height:11px}) only covers the Type/Base Game/Priority row. The general fix is to pin every .fr label to a fixed height (11px, matching the no-sup case) app-wide, with the <sup> positioned to overflow visually without participating in layout height — rather than re-scoping the fix row-by-row again.

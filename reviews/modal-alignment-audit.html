Modal Alignment Audit
Every .fr field across every visibility state of the Add/Edit modal (#mov: Game type, DLC type, and direct-to-Collection add) and the Move-to-Collection modal (#btcov), measured with Playwright/Chromium at 2x DPR — at both a 1280px desktop viewport and a 390px mobile viewport (below the app's 560px breakpoint, where .fg2/.fg3-mid-stretch/.fg4 reflow). Two checks, run everywhere: (1) when two or more fields share a row, do their labels all end at the same Y and their fields all start at the same Y; (2) for every single field independent of row-sharing, is the label-bottom-to-field-top gap the same everywhere.
Box definition (revised): a label's box is the full visual scope of everything rendered as part of it — its own line box, plus any <sup> or trailing note <span>, whichever extends furthest down — not just the parent element's own CSS box, which can be pinned smaller than its content and let that content silently overflow past it. A field's box already includes its border (part of the rendered border-box regardless of box-sizing) and was checked against every hover/focus rule that touches a control used in these rows (.type-btn, .prio-btn, .fetch-btn, .ciw, .ps-modal-btn) — none add an outline, box-shadow, or border-width change on hover/focus, only color/filter, so no state-dependent expansion exists there.
Mixed-control-type rows — where alignment should fall: field top. Gap = fieldTop − labelBottom; if every label in a row ends at the same Y and every field starts at the same Y, the gap is identical for every field by construction, regardless of how tall each individual control is. Bottom-aligning or center-aligning fields would make the gap vary with each field's height. Top-alignment is therefore not just "the exception for mixed rows" but the correct general rule whenever heights differ — it's the only anchor point that preserves a constant gap.

Fix applied: the three failing rows split into two opposite mechanisms — Rows 1 & 6 had unpinned labels that grew taller than 11px whenever a <sup> was present, misaligning the row's fields; Row 2 had a label pinned to 11px whose sup+note content didn't fit and overflowed past the box into the gap. Pinning every label to 11px (the fix originally proposed for Rows 1/6) would have fixed those two while creating Row 2's exact bug everywhere else that has a sup or note.

Instead, style.css's single shared rule .fr label changed from display:block to:

.fr label{display:flex;align-items:flex-start;flex-wrap:wrap;column-gap:.35em;row-gap:.15em;...}

A <sup> or trailing <span> inside a block label shares one inline line box with the main text — vertical-align and mixed font-sizes on that shared line force the whole line taller. As flex items, each run (main text / sup / note) gets its own box on the flex line instead, so vertical-align:super (now a no-op on flex items, and removed from .req) no longer inflates anything, and every label naturally settles at the same ~11px height everywhere — with or without a sup/note — with nothing to pin and nothing to overflow. This fixes both failure mechanisms with one rule change instead of trading one for the other.

Verified by re-running the exact same Playwright audit that found these bugs, post-fix: all 6 desktop row-groups pass, both mobile row-groups pass, and all 118 full-scope per-field gap measurements are exactly 4.000px across both viewports and every modal state (Game/DLC/direct-to-Collection #mov, and #btcov). Screenshots at 4x zoom confirm the MANDATORY tag still reads clearly next to each label, and the Type/Base Game/Priority row now shows the same clean breathing room above its fields as every other row.

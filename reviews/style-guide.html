Live Prices result cards.ggr-grid / .ggr-card
The Check-Live-Prices modal (#ggFetchOv) replaced its monospace log with a dashboard-style grid of result cards — same visual language as the Spend Stats KPI/card tiles (.ss-kpi / .ss-card: var(--base) surface, var(--bd) border, var(--r) radius) rather than the terminal-style log used by the older Release-Date-Check / Fill-Missing-Prices dialogs. Each card shows current retail/keyshop price, a delta badge vs. the price before this run, and either a low €X caption or a "★ new low" badge. Status reads as a 3px left-border accent, same 3-color direction coding used for both the border and the delta badge: green = price dropped, magenta = price rose, cyan = unchanged (pink is reserved for "no data", a fetch failure rather than a price direction) — same "colored stripe = status" idiom as .panel-base-game. The delta/low badges (.ggr-badge) compose with the base .bdg class, so they're pixel-identical in shape to every other badge in the app — only color/text vary.
Cross-device "last results": opening the modal doesn't immediately start a fetch — it first loads the most recent run's results from the sheet (getLatestFetchDiffs, reconstructed from the PriceHistory log) and shows a "Last checked Xh ago" header, so results are visible from any device, not just the one that ran the check. A "Refresh Now" button starts a live run, which reuses the exact same card component as it streams results in. It's disabled ("Loading…") until two independent loads both settle: _ggIdleLoading (this view's own getLatestFetchDiffs call, mostly so a run doesn't start and immediately overwrite the grid mid-populate) and _savedPricesReady (ggPriceCache, populated by loadSavedPrices at boot) — the second one is the actual correctness dependency, since a live run diffs each game against ggPriceCache, not against this view's data; starting a run before it's loaded would show every game as "new" instead of its real delta/low.
Clickable: every card (including "No data" ones) opens that game's side panel on click/Enter — data-appid on .ggr-card, one delegated listener on .ggr-grid so it survives the grid's innerHTML being replaced every batch/range-switch without re-binding per card. Always *hides* the modal rather than closing it, whether or not a run is active — closing would replace its history entry, so Back from the panel landed on the bare main view instead of returning here. Hiding leaves the entry and the rendered grid intact; the panel-close branch of the app's main popstate handler re-shows the modal when Back lands back on that entry. Hover/focus swaps the card's background to var(--s1) — the left-border status accent is intentionally left untouched by the hover state, not overridden.
Filter row .ggr-filter-row: All / Down / Up over the cards below — these are real Filter Pills (.fbar-pill, the same component as the sidebar's Priority/Platform/Play status/Hotness filters, documented in the Pill/Badge/Chip reference), not a Selector Pill like .plat-tab/.ph-range-pill — a filter pill's own color is fixed and always visible (blue/green/magenta here) rather than only coloring in when selected, opacity alone carries selected state (.55 → .8 hover → 1 selected, identical to every other Filter Pill), and each always shows a live count via .fbar-pill-count. Down/Up don't recompute anything — they just hide cards whose existing .ok/.up status class doesn't match, via .ggr-grid[data-filter="…"] .ggr-card:not(.…){display:none}, so a live run's newly-appended cards are filtered (and counted) automatically with no extra JS per batch. "Unchanged" and "No data" cards only count/show under All. Resets to All (and the row hides entirely) each time the modal loads a fresh view — opening the idle "last results" load, or starting a new run via Refresh Now — but stays as-is while a run's batches stream in, so picking Up mid-run keeps showing (and counting) only Up as more results arrive. A small .ggr-empty message appears if the current filter matches zero cards.

Next Up picker#nuov / .nu-stage
Menu item (Hamburger menu → "Next up", both mobile #hmenu and desktop #dhmenu) that picks one random game from whatever the Collection filter bar currently shows — collectionFiltered(), minus DLC already nested under a parent. Deliberately has no picker-owned weighting (by priority, hotness, staleness, etc.): narrowing the candidate pool is the filter bar's job (Play status / Genre / Platform / Steam collection), not a second filtering system layered on top.
Reveal animation .nu-shuffle-face: a slot-machine-style cycle through random candidates' cover art (reusing the real .cc/.cph cover markup, so a missing cover falls back to the same placeholder as everywhere else) plus title (.ct, identical to the card title style). Each tick is a fresh DOM node so the nuTick keyframe (scale .96→1, opacity .55→1) restarts automatically — no manual reflow trick needed. Ticks decelerate via a quadratic ease (50 + 260*(i/n)² ms), landing after ~12–16 ticks on open or ~7–10 on Reroll.
Landed card .nu-result: the actual pick renders via colCardHTML() — the same function the Collection grid uses — inside a single-item .gg wrapper, so it is pixel-identical to its grid counterpart (play-status badge, platform badges, quick actions, everything) rather than a bespoke summary card. A short nuLand pop-in (scale .88→1) plays once on arrival. Clicking the card (outside its quick-action icons, which already stop propagation) closes the picker and opens that game's side panel, same as clicking it in the grid. Reroll re-runs a shorter shuffle and avoids repeating the immediately-previous pick when the pool has more than one game; it hides itself when the pool is exactly one game, since there's nothing else to land on.
Empty state .nu-empty: plain centered text (no icon) when the current Collection filters match zero eligible games, telling the user to adjust the filter bar rather than the picker offering its own fallback logic.

Fetched once on panel open from the PriceHistory sheet (every live-price check is logged there); switching the range filter re-slices that same fetched set client-side rather than re-hitting the network. Hand-rolled inline SVG, same idiom as the Spend Stats trend chart (.ss-line-*): retail (var(--cyan)) and keyshop (var(--amber)) lines with dot markers, a dashed reference line at each series' personal-low, hover/focus tooltip + crosshair via invisible hit rects. Needs at least 2 recorded price checks (in the selected range) to draw a line; shows an empty-state message otherwise, with the range row still visible so you can back out to a wider range.
Tooltip clamping: the tooltip is center-anchored on its data point by default (transform:translate(-50%,-125%)) but its left is clamped to [halfWidth, containerWidth − halfWidth] before being set, so it never renders partway outside .ph-chart's own box — the first/last points used to push half the tooltip past the panel edge, where it got clipped by the panel's scroll container instead of just repositioning. The tooltip also shows the fetch time (D Mon, HH:mm, e.g. "10 Jul, 09:18"), not just the date, since multiple checks on the same day are otherwise indistinguishable. Axis labels use text-anchor: start/end (not always middle) at the first/last tick for the same reason — they grow inward from the edge instead of centering on it and clipping.
Range filter .ph-range-row / .ph-range-pill: 7D / 1M / 3M / 6M / 1Y / All, defaulting to All. Same selector-pill shape/grammar as .plat-tab (20px radius, 1.5px border, transparent at rest, solid var(--blue) fill + dark text when .on) — picking a history window is the same kind of one-of-many choice as picking a platform tab. Wraps to a second line on narrow panels instead of overflowing.
